# Agent Harness v0.8.0

Agent Harness `0.8.0` integrates durable repository control with Codex-native
Goal, Plan, Thread, and subagent execution while making the default experience
lighter.

## Three Execution Paths

- `codex-direct`: ordinary clear work uses Codex and creates no Harness
  lifecycle artifacts.
- `codex-direct-postflight`: simple work linked to existing Harness state uses
  Codex execution followed by one bounded verification and state-sync closeout.
- `durable-harness`: recovery, audit, milestone/DAG, multiple workers,
  persistent state sync, or high-risk work retains repository Goal/Run control.

## Codex-Native Bridge

- Runtime Goal owns the current long-running outcome and continuation.
- Codex Plan owns current steps and transient progress.
- Codex runtime owns Thread/subagent scheduling, concurrency, cancellation, and
  model/effort selection.
- Repository Goal/Run owns cross-task recovery, durable dependencies and
  ownership, acceptance evidence, Delivery State, and bounded state sync.

Controller now means outcome owner and accepted-state owner. It may implement
foreground work; only explicit `gate-only` or review-only direction makes the
controller non-editing.

## Durable Boundaries

- Configured completion gates apply to durable Goal/Run completion, not to
  ordinary direct work or postflight-only synchronization.
- A prepared enforced Run cannot be downgraded to postflight-only to bypass its
  DAG, checklist, gates, or evidence.
- Postflight updates existing tracked state only and never creates lifecycle
  artifacts solely for bookkeeping.
- Native Goal/Plan fallback continues in the current thread without inventing
  runtime identifiers; degraded provenance is recorded only when a durable Run
  requires it.
