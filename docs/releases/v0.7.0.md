# Agent Harness v0.7.0

Agent Harness `0.7.0` narrows the product around the persistent project control
that current Codex runtime capabilities do not replace.

## Highlights

- Keeps deterministic config and doctor checks, adapter-declared paths,
  repository Goal/Run/DAG state, delivery evidence, and controller-owned gates.
- Returns ordinary clear change/build execution, worker scheduling,
  concurrency, cancellation, and model choice to the Codex runtime.
- Removes the public `mixed` role and the bundled explorer/implementer agent
  templates. The optional reviewer remains read-only and inherits the parent
  session model and reasoning effort.
- Reduces route, rule, reference, template, and documentation duplication while
  preserving durable recovery, audit, milestone, DAG, and state-sync behavior.

## Safety And Correctness

- Configured paths reject absolute paths and traversal outside the project.
- Goal, Spec, Run, DAG node, status, result, and log paths are contained in
  their configured roots, including existing symlink targets.
- Run preparation records the starting branch, HEAD, upstream, and dirty-state
  digest. Completion derives committed and pushed evidence from that Run-scoped
  snapshot rather than from the repository's current global state alone.
- The local marketplace is named `agent-harness-local`; deployment validates
  marketplace, plugin, source root, selector, and cache identity before refresh.

## Compatibility And Migration

- Config readers continue to accept supported legacy aliases, but generated
  config writes canonical fields only. Conflicting old and new values fail
  validation instead of choosing one silently.
- Removed config fields that were not consumed by the CLI are no longer emitted.
- `harness:intake` no longer uses implicit invocation. Ordinary change/build
  requests do not enter the durable Harness path unless recovery, audit,
  milestone/DAG, multi-worker, persistent state sync, or higher risk requires it.
- Plugin `interface.defaultPrompt` now uses the array form.

## Verification

```bash
npm run test:all
npm run test:eval
npm run validate:plugin
git diff --check
```

The current Codex skill validator also passes for `orient`, `intake`, `init`,
and `execute`.

The deterministic routing-classification suite validates fixture routes and
CLI/trace contracts. It does not claim real plugin activation provenance.

## Delivery Notes

The source tree and release documentation can be validated locally without
claiming a tag, GitHub Release, push, deployment, or refreshed plugin cache.
Those delivery steps require separate explicit authorization and evidence.
